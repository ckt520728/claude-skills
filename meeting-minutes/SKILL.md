# Meeting Minutes

## 目的

把雜訊很多的會議逐字稿,變成**三週後翻出來還看得懂、且每條 action 都能追蹤**的會議紀錄。

原始逐字稿的問題:資訊密度低、講者反覆、有口頭禪、決議與閒聊混在一起。這個 skill 解決的就是「會後我到底要追什麼?哪些是決定?哪些只是討論?誰要做?何時做?」

## 觸發後的工作流程

### Step 1:確認輸入來源

判斷或詢問:

1. **逐字稿位置**:
   - 對話貼進來 → 直接處理
   - 檔案路徑(`.txt` / `.md` / `.vtt` / `.srt` / `.docx`)→ Read
   - 多個檔案(例如連續幾場會議)→ 逐一處理,各自輸出
2. **會議基本資料**(若逐字稿裡沒寫,要主動問):
   - 會議日期(沒寫 → 預設今天)
   - 會議主題 / 目的
   - 出席者(沒寫 → 從逐字稿的講者標籤推斷,推不出來就直接寫「未指明」)

**例外**:如果使用者一次給很多檔案且明確說「批次處理」,別逐場確認,用預設值跑完,在最後一次列出「以下會議缺少日期/出席者資訊,需要補」。

### Step 2:預讀逐字稿,做語意分類

把逐字稿的每段話在腦中分到以下五類(**不要把這個分類過程輸出給使用者**):

| 類別 | 標記 | 範例觸發 |
|------|------|---------|
| **決議**(Decision) | D | 「我們就這樣定了」「OK 那就 X」「下次起改用 Y」 |
| **行動項目**(Action) | A | 「X 你下週前處理一下」「我來寫一份草稿」 |
| **未解問題**(Open Q) | Q | 「這個還要再確認」「之後再討論」「parking lot」 |
| **重要討論**(Discussion) | T | 不是決議也不是 action,但對理解上下文重要 |
| **雜訊**(Noise) | N | 開場寒暄、設備測試、岔題閒聊、口頭禪重複 |

只有 D / A / Q / T 會進到輸出,N 全部丟掉。

### Step 3:產出結構化 Markdown

固定七個區塊。**空區塊整段刪掉**,不要寫「(無)」充版面。

```markdown
# 會議紀錄:<會議主題>

- **日期**:<YYYY-MM-DD>
- **時間**:<HH:MM - HH:MM>(若逐字稿沒寫就刪掉)
- **出席者**:<姓名 1>, <姓名 2>, ...
- **缺席 / 應到未到**:<姓名>(若無就刪掉)
- **紀錄者**:<AI 整理,基於 <逐字稿檔名>>

---

## 1. TL;DR(30 秒讀完)

<3-5 句。讓沒到場的人讀完就知道:會議結論是什麼、誰要做什麼、下次會議聚焦什麼。**不要**列點,寫成段落。>

## 2. 決議事項(Decisions)

> 決議 = 已經拍板、不會再回頭討論的事。如果使用者在後段又翻案,以最後一次為準。

1. **<決議的一句話標題>**
   - 內容:<2-3 句具體決議內容>
   - 理由:<為什麼這樣決定——三週後翻紀錄時最重要的就是這行>
   - 反對 / 保留意見:<若會議中有人明確反對,記下來;沒有就刪這行>

2. **<下一個決議>**
   - ...

## 3. 行動項目(Action Items)

| # | 行動 | 負責人 | 期限 | 完成標準 |
|---|------|--------|------|---------|
| 1 | <具體做什麼,動詞起頭> | <姓名> | <YYYY-MM-DD 或 "下次會議前"> | <怎樣算做完> |
| 2 | ... | ... | ... | ... |

**強制規則**:
- 每一條 action 都**必須**有負責人。逐字稿沒講清楚 → 寫 `?? (需確認)` 並在 TL;DR 提醒
- 每一條 action 都**必須**有期限。沒講 → 寫 `?? (需確認)` 並提醒
- 「完成標準」要可驗證(例如「寄出 PR 連結到群組」),不能寫「持續關注」「思考一下」

## 4. 未解問題 / Parking Lot

- <問題 1>(誰提的、為什麼還沒解決)
- <問題 2>

## 5. 重要討論摘要

> 不是決議、也不是 action,但對理解後續決策很重要的脈絡。

### <討論主題 1>
<2-4 句摘要,保留關鍵的不同立場。如果有人說了特別關鍵的話,直接 quote。>

> "<原話引用>" —— <講者>

### <討論主題 2>
...

## 6. 下次會議 Carry-Over

下次會議應該優先討論的事:

- [ ] <第 3 區動 action 中,期限是「下次會議前」的項目>
- [ ] <第 4 區未解問題中,需要追蹤的>
- [ ] <逐字稿中明確說「下次再討論」的議題>

## 7. 附註 / 風險

<會議中提到、但不適合放上面的內容。例如:時程風險、人力風險、外部依賴、需告知缺席者的事。沒有就刪掉這區塊。>
```

### Step 4:寫檔 + 簡短回報

輸出檔名:`meeting_<YYYY-MM-DD>_<topic-slug>.md`,slug 從會議主題轉成英文/拼音 kebab-case。

預設寫到逐字稿所在資料夾;使用者指定其他位置就照辦。

回報只說:
- 路徑
- Action items 共 N 條,其中 M 條缺 owner / K 條缺 deadline(若有缺,**列在 chat 裡讓使用者立刻補**)
- TL;DR 第一句

**不要**把整份紀錄再貼一次。

## 重要的編輯原則

**Decision vs Discussion 嚴格區分**:這是這個 skill 跟「逐字稿摘要」最大的差別。判定標準:

- 是 Decision 嗎?問自己:「會後有人問起,大家會說『這個我們會議裡定了 X』嗎?」
- 還是 Discussion?「我們會議裡討論過,但最後沒有結論」

如果在邊界灰色地帶,放到第 5 區(重要討論)+ 第 4 區(未解問題),**不要**寫成 Decision。錯把 Discussion 寫成 Decision,後果是團隊以為已經拍板、實際上沒有,會出大事。

**保留決策理由**:Decision 區塊的「理由」欄是最容易被偷工的——但這欄正是會議紀錄的最大價值。三週後翻紀錄,「我們決定用 X」沒用,「我們決定用 X 因為 Y 客戶下季要 launch」才有用。如果逐字稿裡找不到明確理由,**就寫「逐字稿未明確說明」**,不要瞎掰一個理由。

**Action 完成標準必須可驗證**:
- ❌ 「思考一下下一步」
- ❌ 「持續追蹤」
- ❌ 「再研究看看」
- ✅ 「寄出 RFC 草稿到 #planning 頻道」
- ✅ 「跑完 benchmark 並貼結果在這份紀錄底下」

如果逐字稿裡的 action 真的就是空泛的「想一下」,**不要美化它**——照實寫「(待釐清:這條 action 在逐字稿中未指定具體交付物)」並在 TL;DR 提醒。

**人名要對**:逐字稿如果用 `Speaker 1` / `講者 A`,**不要**自己編一個名字。直接保留 `Speaker 1`,或如果使用者有提供出席者名單,在能高信心對應時才換成真實姓名。錯把張三的話安在李四頭上比沒記名字更糟。

**保留原始語言**:逐字稿是中文 → 紀錄用中文。是英文 → 用英文。中英混雜(台灣會議很常見)→ 紀錄也混雜,**不要**強制翻譯統一語言。引用原話時尤其要保留原語言。

**敏感資訊要敏感**:逐字稿中若出現以下內容,在輸出時:
- 個人身分資料(身分證號、生日、住址)→ 遮罩(只留首尾,中間用 *)
- 薪資 / 內部數字 → 保留但在檔案開頭加 `> ⚠️ 此紀錄含內部敏感資訊,請勿外流`
- 對其他人的負面評價 → 改寫成中性陳述,或移到第 7 區「附註」並標明來源

## 輸出檔案的編碼

UTF-8 無 BOM。Windows PowerShell 環境用 Write 工具或:

```powershell
$utf8NoBom = New-Object System.Text.UTF8Encoding($false)
[System.IO.File]::WriteAllText($path, $content, $utf8NoBom)
```

## 不要做的事

- ❌ 不要呼叫外部 LLM API——Claude 自己整理就好,不要多一層失真
- ❌ 不要把口頭禪(「然後就是」「對對對」「ok ok」)留在紀錄裡
- ❌ 不要按時間順序逐段流水帳——按**語意**重新組織(決議放決議區、行動放行動區),不是按發言順序
- ❌ 不要寫「會議圓滿結束」「大家討論熱烈」這種空話
- ❌ 不要把逐字稿原文整段貼進來——只在引用關鍵原話時用 blockquote,且要短(< 2 行)
- ❌ Action 條目超過 15 條是警訊——大概率你把 discussion 也當 action 了,回去重看
- ❌ 不要刪掉 `?? (需確認)` 標記去「美化」紀錄,缺資訊就是缺資訊,讓使用者知道才能補
