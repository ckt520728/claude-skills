# Research Organizer

## 目的

把使用者餵進來的研究素材(可能是貼在對話裡的長文、TXT 檔、PDF/DOCX 抽出的文字、會議錄音逐字稿、論文摘錄、隨手記下的研究想法),整理成一份結構清楚、可後續行動的 Markdown 研究筆記。

這是給「自己長期累積研究/閱讀素材」用的工具,不是給別人讀的 publication——重點是**抓住可重用的內容**(金句、可行動洞察、技術概念、待釐清問題),不是漂亮排版。

## 觸發後的工作流程

### Step 1:確認素材來源

詢問或自行判斷使用者是哪一種情況:

1. **直接貼在對話裡**:文字已在 context,直接進 Step 2
2. **指定檔案路徑**(`.txt` / `.md` / `.pdf` / `.docx`):用 Read 工具讀取,PDF/DOCX 視需要呼叫對應 skill
3. **指定資料夾**:用 Glob 列出所有相關檔案,逐一讀取後合併
4. **多個來源混合**:全部讀進來,在最後輸出時保留來源標記

如果使用者沒指定輸出位置,預設寫到**素材所在資料夾**,檔名為:
- 單一來源:`<原檔名>_organized.md`
- 多個來源:`research_summary_<YYYYMMDD>.md`

### Step 2:分析素材

讀完素材後,在腦中(不要直接輸出給使用者)做以下分析:

- **主題定位**:這份素材在講什麼?屬於哪個研究領域/問題?
- **核心論點**:作者/講者想證明或主張什麼?
- **證據與方法**:用什麼資料、實驗、論證來支持?
- **未解問題**:作者明確說「還沒解決」「未來工作」的部分
- **可遷移性**:哪些想法可以套用到使用者自己的研究/工作?

### Step 3:產出結構化 Markdown

固定產出以下七個區塊。如果某個區塊在素材裡完全找不到內容,就**整段拿掉**,不要寫「(無)」充版面。

```markdown
# <主題>:<一句話的研究定位>

> 來源:<檔名 / URL / 「使用者口述」>
> 整理日期:<YYYY-MM-DD>

## 1. 一段話摘要

<3-5 句話,讓三個月後的自己讀完就能想起整篇在說什麼。不要超過 150 字。>

## 2. 核心論點(Key Ideas)

- **<論點 1 的單句標題>**:<2-3 句解釋,包含為什麼這個論點重要>
- **<論點 2 的單句標題>**:...

(通常 3-7 點。每點都要能獨立存活——只看這一行也讀得懂。)

## 3. 重要技術概念 / 名詞

| 概念 | 定義 | 為什麼重要 |
|------|------|-----------|
| <term 1> | <一句話定義> | <在此素材的上下文裡為什麼出現> |

(如果素材裡沒有明顯的「新術語/技術概念」,整個區塊刪掉。)

## 4. 可行動洞察(Actionable Insights)

針對使用者「之後可以拿來做什麼」的具體建議。每條都要回答「下一步具體做什麼」,不要寫成空泛的感想。

- [ ] <具體行動 1>(例如:把 X 方法套用到 Y 資料集,看是否 reproduce)
- [ ] <具體行動 2>

## 5. 引用價值的金句 / 數據

> "<原文段落或數據>"

—— <出處頁碼/段落>,<一句話說明為什麼這段值得記下來>

(2-5 條為佳。沒有就刪掉這個區塊。)

## 6. 未解問題 / 待釐清

- <問題 1>
- <問題 2>

(作者明說、或使用者讀完後可能想追問的問題。)

## 7. 與其他研究的連結

- 與 <X 主題/作者/論文> 的關係:...
- 可延伸閱讀:...

(如果素材本身沒提到 related work,且使用者沒提供其他來源,可以省略。)
```

### Step 4:寫檔 + 簡短回報

用 Write 工具寫到 Step 1 決定的路徑。回報時:

- 只說「已寫到 `<path>`」+ 一句話總結最重要的 1-2 個發現
- **不要**把整份 markdown 內容再貼一次給使用者(他自己會去開檔案)
- **不要**在 chat 裡逐項覆述每個區塊

## 重要的編輯原則

**節錄勝過改寫**:遇到關鍵句、定義、數據,**直接引用原文**並標來源,不要自己改寫。改寫會引入失真,三個月後翻筆記時無法判斷是作者原意還是 Claude 的詮釋。

**抽象度配對**:使用者餵的是粗糙素材 → 產出要結構化但仍保留細節;使用者餵的是已經精煉過的摘要 → 不要再過度濃縮成更短的摘要,而是補上「可行動洞察」與「未解問題」這類延伸層。

**Actionable 區塊不可空泛**:第 4 區塊每一條都要是「明天能動手做」的事。如果只能寫出「持續關注 X」「思考 Y 的應用」這種空話,寧可不寫——讓區塊變短或整段刪掉。

**保留使用者語言**:使用者用什麼語言貼素材,輸出就用什麼語言。中文素材輸出中文筆記,英文素材輸出英文筆記。**不要**自動翻譯,除非使用者明確要求。

## 輸出檔案的編碼

寫檔一律用 UTF-8(無 BOM)。如果環境是 Windows PowerShell,**不要**用 `Out-File` / `Set-Content` 預設的 UTF-16 LE,改用 Write 工具或:

```powershell
$utf8NoBom = New-Object System.Text.UTF8Encoding($false)
[System.IO.File]::WriteAllText($path, $content, $utf8NoBom)
```

## 不要做的事

- ❌ 不要呼叫外部 LLM API(OpenAI / Gemini 等)——本技能就是 Claude 在做,呼叫外部反而多一層失真
- ❌ 不要產出「七個區塊但每區塊只有一行」的空殼筆記。寧可只交三個有內容的區塊
- ❌ 不要在筆記頂端加 emoji 或 ASCII art 裝飾
- ❌ 不要在筆記裡寫「本筆記由 AI 整理」這類元資訊
- ❌ 不要把使用者的素材改寫成第三人稱論文體——保留筆記的「給自己看」語氣
